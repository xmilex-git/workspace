# PR 4869: [CBRD-24046] Result cache for correlated scalar subquery

- state: MERGED / base: develop / created: 2024-01-11
- diff: +1164 -7, 18 files

## Code changes (summary)

- New files `src/query/subquery_cache.{c,h}` implementing a per-XASL correlated-scalar-subquery result cache (`SQ_CACHE`) keyed on the subquery's bound host-variable/correlated-column values, backed by an `MHT_TABLE` hash table with hit/miss stats and a byte-size cap
- New hidden system parameter `max_subquery_cache_size` (`PRM_SIZE_UNIT`, default 2MB, range 0–16MB) added to `system_parameter.{c,h}`
- New XASL flag `XASL_USES_SQ_CACHE` and a `sq_cache` field added to `xasl_node` (`xasl.h`); packed/unpacked for client-server transfer via new `xts_process_sq_cache()` in `xasl_to_stream.c`
- New query hint `NO_SUBQUERY_CACHE` added end-to-end (grammar in `csql_grammar.y`, bit flag in `parse_tree.h`, hint parsing in `scanner_support.c`, printing in `parse_tree_cl.c`) to let a query opt out of caching
- `xasl_generation.c`: new `pt_prepare_corr_subquery_hash_result_cache()` determines whether a correlated scalar subquery is cacheable and builds its cache-key structure; new `pt_check_corr_subquery_not_cachable_expr()` walker rejects non-deterministic/side-effecting subquery expressions
- `query_dump.c`: cache hit/miss/size statistics added to both JSON and text query-trace output
- Build files (`cubrid/CMakeLists.txt`, `sa/CMakeLists.txt`) and `.github/CODEOWNERS` updated to register the new `subquery_cache.c` source; `cubrid-jdbc` submodule pointer bumped
- `fetch.c`/`query_executor.c`/`stream_to_xasl.c` updated to look up/populate cache entries at subquery evaluation time (get on evaluate, put on first compute)

## PR body (verbatim)

http://jira.cubrid.org/browse/CBRD-24046

A correlated subquery is executed for each row of the main query to get a value. If there are many duplicated inputs, the same query is repeatedly executed, which may cause a decrease in performance. We eliminate the cause of the slowdown through the result cache.
