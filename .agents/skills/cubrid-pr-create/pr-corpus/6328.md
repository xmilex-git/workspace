# PR 6328: [CBRD-26194] Parallel query execution worker 에서 IS_LOCK을 요청 시 thread-safe하게 수정

- state: MERGED / base: develop / created: 2025-07-21
- diff: +59 -3, 8 files

## Code changes (summary)

- Added a new `pthread_mutex_t m_px_lock` member to `cubthread::entry` (src/thread/thread_entry.hpp/.cpp), initialized/destroyed alongside the other thread-entry mutexes, and reset `m_px_orig_thread_entry` to NULL on task cleanup (src/thread/thread_entry_task.cpp).
- In `px_heap_scan_manager.cpp` and `px_query_executor.cpp`, set `thread_p->m_px_orig_thread_entry = thread_p` on the originating thread when starting a parallel heap scan / parallel query executor; moved the equivalent assignment in `px_query_task.cpp` so it runs unconditionally (not only when `is_on_parallel_worker`).
- In `qexec_prune_spec` (src/query/query_executor.c) and `heap_scancache_start_internal` (src/storage/heap_file.c), wrapped the `lock_subclass`/`lock_scan (..., IS_LOCK)` calls in `#if defined(SERVER_MODE)` guarded lock/unlock of the originating thread's new `m_px_lock` mutex, so concurrent parallel workers sharing the same `tran_index`/`class_oid` no longer race on the lock manager.

## PR body (verbatim)

<http://jira.cubrid.org/browse/CBRD-26194>

### Purpose

현재 lock manager는 동일한 tran_index와 class_oid를 가진 lock 요청이 동시에 수행될 때 thread_safe 하지 않다. 이러한 동시 요청은 parallel query execution에서 수행된다. 따라서 parallel query execution 에서의 lock 요청 시 별도의 mutex를 잡게끔 변경한다.
