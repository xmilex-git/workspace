# PR 4833: [CBRD-25067] ORDER BY causes row to be omitted with index and case expression

- state: MERGED / base: develop / created: 2023-12-07
- diff: +4 -2, 2 files

## Code changes (summary)

- `qo_search_isnull_key_expr()` (`query_planner.c`): replaced the hardcoded check for `PT_IS_NULL`/`PT_IS_NOT_NULL`/`PT_IFNULL`/`PT_NULLSAFE_EQ` operators with a call to `qo_check_nullable_expr()` that walks the full key-expression subtree and counts nullable terms; the branch now triggers when `nullable_terms >= 1`
- `qo_check_nullable_expr()` (`query_rewrite.c`): added `PT_NULLSAFE_EQ` to the set of operator cases treated as potentially nullable
- Net effect: index scan is no longer chosen for ORDER BY / GROUP BY plans when the key expression contains any nullable term, not just the previously-hardcoded set of ops

## PR body (verbatim)

http://jira.cubrid.org/browse/CBRD-25067

When using order by, an error occurred where null values did not appear in the results even though expressions capable of null input existed as terms in the query, and an index scan was used. 

The revised code is a function that checks for the presence of nullable key expressions before creating order by plans. 
- It has been modified to not use index scan in order by or group by plans if there are nullable terms in the key expressions, by using the function qo_check_nullable_expr() to check for nullable terms in key_expr.
