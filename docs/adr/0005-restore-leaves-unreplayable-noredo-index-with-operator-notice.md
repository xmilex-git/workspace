# ADR-0005: 복원은 항상 완료하고, 재생 불가 no-redo 인덱스는 운영자 안내로 넘긴다 + no-redo는 병렬 빌드 한정

날짜: 2026-07-19. 상태: **ADR-0006으로 대체됨(2026-07-20)** — loaddb 한정 피벗으로 "복원은 항상
완료 + 운영자 안내" 정책 자체가 폐기되고, 완결된 no-redo 빌드 구간의 replay는 명시 에러로
거부된다(불완전 인덱스 상태가 도달 불가; 병렬 한정 강등은 ADR-0006에 승계). 이전 상태:
승인(보스 결정). 관련: ADR-0002, ADR-0003, ADR-0004, ADR-0006, CBRD-27071.

## 맥락

no-redo 빌드 인덱스의 페이지는 WAL에 없다. 빌드 이전 백업 + full replay 복원에서는 해당
인덱스가 재생 불가이므로, 기존 설계는 restoredb 말미에 marker 트랜잭션의 발행 구간 로그를
물리 역재생(publication walk)해 카탈로그에서 제거했다.

검증(R1 레인)에서 확인된 사실: 발행 이후 같은 클래스에 DDL이 한 번이라도 있으면(클래스
표현 레코드는 DDL마다 삭제+재삽입으로 교체됨; vacuum replay도 동일 효과) walk의 물리 undo
대상 슬롯이 이미 소멸해 ER_SP_UNKNOWN_SLOTID로 실패하고, 안전 가드가 **복원 전체를 FATAL**
시킨다. 이는 "빌드 전 백업 → 이후 동일 테이블 DDL → full-replay 복원"이라는 평범한 운영
시퀀스에서 재현되므로 가용성 문제다. 물리 역재생은 발행 구간과 tip 사이에 개입 쓰기가
있으면 원리적으로 건전할 수 없고(부재 슬롯을 관용해도 이후 표현 레코드에 남은 인덱스
참조는 제거 불가 → 반쪽 카탈로그), 카탈로그 레코드의 선택적 redo 스킵도 공유 페이지의
물리 연속성을 깨뜨려 불가하다.

## 결정

1. **restoredb/replay는 항상 완료한다.** 발행 구간이 깨끗한 경우(개입 쓰기 없음)에만 기존
   publication walk로 카탈로그에서 제거한다(현행 유지). 개입 쓰기가 감지되면(사전 read-only
   검증 패스에서 undo 대상 부재 발견) walk를 수행하지 않고 카탈로그·파일을 그대로 두며,
   restoredb 말미에 운영자 안내를 출력한다:
   "index <name> on <class> was built without redo logging and cannot be replayed from this
   backup; drop and re-create it before use: DROP INDEX <name> ON <class>; using it without
   rebuilding causes errors."
2. **안내를 무시하고 해당 인덱스를 사용하다 발생하는 오류는 사용자 책임으로 한다.**
   (인덱스 페이지는 미복원 상태이므로 사용 시 오류가 발생할 수 있다. DROP INDEX는 파일
   헤더/섹터 메타데이터만 사용하므로 페이지 미복원과 무관하게 정상 동작한다.)
3. **no-redo 빌드를 병렬 빌드 경로가 실제로 관여하는 경우로 한정한다.** 직렬 경로(임계
   미달, n_shards<2 폴백, 빈/소형 테이블)는 eligible이어도 전체-로깅 빌드로 강등한다.
   강등 시 marker 미발행·pending 미등록·클라이언트에 NULL create LSA 반환(ADR-0004의
   강등 기제 재사용). 직렬 빌드는 어차피 소규모라 no-redo 이득이 없고, 이 축소로 marker
   노출 표면과 복원 안내 발생 빈도가 함께 줄어든다.

## 대안과 기각 사유

- 부재 슬롯 관용 후 walk 완주: 이후 표현 레코드의 인덱스 참조를 제거하지 못해 반쪽
  카탈로그(조용한 불일치) — 기각.
- restoredb 말미 SA-클라이언트 자동 DROP(논리 제거): 유일하게 자동+일관을 만족하는 완성형.
  유틸리티 내 클라이언트 부팅 사이클 설계가 필요해 이번 게이트 범위를 초과 — **후속 과제로
  유지**(승격 시 본 ADR의 안내 출력이 자동 드랍으로 대체된다).
- 루트 페이지 poison(오류 결정론화): 안내 무시 시의 오류 형태를 깔끔하게 만들지만, 본
  결정은 그 경우를 사용자 책임으로 정의하므로 필수 아님 — 채택 안 함.

## 결과

- 복원 가용성 회복: churn 여부와 무관하게 restoredb 성공.
- fail-safe 유지: 어떤 경로에서도 조용한 카탈로그 불일치를 만들지 않는다(깨끗한 창 =
  완전 제거, 오염된 창 = 완전 보존 + 명시 안내).
- walk의 FATAL 가드는 "깨끗한 창에서의 예기치 못한 실패" 한정으로 존치.
